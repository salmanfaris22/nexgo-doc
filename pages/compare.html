NexGo vs Next.js

N

NexGo

  • ✓ No Node.js, no npm, no runtime
  • ✓ Single ~10MB binary
  • ✓ 200,000+ requests per second (cluster mode)
  • ✓ Go's type safety and performance
  • ✓ Zero external dependencies
  • ✓ Built-in auth, sessions, CSRF
  • ✓ WebSockets, ORM, plugin system
  • ✓ Islands Architecture — 0 KB JS by default
  • ✓ 5 hydration strategies (load/visible/idle/media/none)
  • ✓ Deploy anywhere Go runs
  • ✗ No React ecosystem
  • ✗ Smaller community (for now)
N

Next.js

  • ✓ Huge React ecosystem
  • ✓ Large community & resources
  • ✓ TypeScript first-class support
  • ✓ Rich component ecosystem (shadcn, etc.)
  • ✗ Requires Node.js runtime
  • ✗ Heavy node_modules (~300MB+)
  • ✗ Slower cold starts
  • ✗ ~85KB JS minimum per page (React runtime)
  • ✗ Full-page hydration — no per-component control
  • ✗ Lower throughput vs Go

💡 Why Islands? Every Next.js page ships ~85KB of React runtime even for a static blog post. With NexGo Islands, static pages ship 0 KB of JavaScript. Only interactive components load JS — and you control when with 5 hydration strategies.

When to use NexGo

✓ Use NexGo when...

  • → You want Next.js DX with Go performance
  • → You want zero-JS pages with Islands Architecture
  • → You're building a content site, blog, or docs
  • → You want a single binary with no Node.js
  • → You need SSR + SSG + API in one framework
  • → You want zero external dependencies
  • → You deploy to resource-constrained servers
  • → Your team knows Go

✗ Consider alternatives when...

  • → You need React components and ecosystem
  • → Your team only knows JavaScript/TypeScript
  • → You need a complex SPA with rich UI
  • → You need maximum raw routing performance
  • → You need native mobile app support
